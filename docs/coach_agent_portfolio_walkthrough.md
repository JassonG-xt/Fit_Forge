# Coach Agent Portfolio Walkthrough

A reviewer-facing walkthrough of FitForge's Coach Agent. It exists to help an
external reader (recruiter, peer reviewer, fellow engineer) understand **why
this project is not just an LLM wrapper** without having to read every file
under `lib/agent/`, `agent_backend/agents/`, and `docs/`.

This document is a positioning artifact, not a feature spec. The canonical
phase narrative lives in
[`docs/recovery_routing_phase_summary.md`](recovery_routing_phase_summary.md);
the canonical capability map lives in
[`docs/agent_capabilities.md`](agent_capabilities.md). If anything below
disagrees with code under `lib/`, `test/`, `agent_backend/`, or
`.github/workflows/`, the code wins.

## 1. Project positioning

FitForge is a local-first Flutter fitness planner with a bounded Coach Agent
layer. The LLM is treated as an untrusted intent parser that emits structured
actions; all state mutations are previewed, user-confirmed, hash-guarded, and
executed deterministically through local domain engines.

In Chinese (to match the in-app voice):

> LLM 只提出结构化建议；用户确认；本地确定性 executor 执行。

What this implies in practice:

- The LLM never writes `AppState`.
- The backend never writes `AppState`.
- Every plan-mutating action is structured data the user must explicitly
  confirm in the Flutter UI before any local write happens.
- Recovery-aware coaching adds *routing* and *suggestion text*, not new
  mutation paths.

## 2. Architecture at a glance

```
User natural language
  → AgentService / backend or mock client
  → structured AgentAction
  → validation / sourceContextHash checks
  → AgentActionCard preview
  → user confirmation
  → LocalAgentActionExecutor
  → AppState / deterministic engines
```

Three notes that matter for understanding the surface:

- **Mock mode is the deterministic baseline.** `FITFORGE_AGENT_MODE=mock` runs
  fully offline with a keyword router; this is the CI baseline and the default
  demo path.
- **Real mode is optional / experimental.** The backend talks to an
  OpenAI-compatible endpoint only when explicitly configured. The real path
  has manual smoke evidence, not a production-readiness claim.
- **The backend is an LLM adapter, not the source of user state.** All user
  state (profile, plan, sessions, history) lives in local
  `SharedPreferences`; the backend only relays structured suggestions and
  re-stamps safety fields.

Full per-step diagrams (data flow, mutation safety boundary, safety
short-circuit, generatePlan boundary, eval/CI boundary) live in
[`docs/agent_architecture_diagram.md`](agent_architecture_diagram.md).

## 3. Local-first deterministic core

Underneath the Agent layer, the app is a deterministic fitness planner:

- **`AppState`** (`lib/services/app_state.dart`) is the in-memory source of
  truth.
- **`SharedPreferences`** handles local persistence; there is no cloud
  storage, no account, no sync.
- **`PlanEngine`** auto-generates weekly workout plans from the user profile
  (frequency, goal, experience, equipment, split family).
- **`NutritionEngine`** computes BMR → TDEE → target calories, macros, and
  water from the same profile.

The LLM does **not** replace these engines. When the user asks "give me a
plan," the Agent emits a `generatePlan` action; the actual preview is
generated by `PlanEngine` deterministically; the optional Agent preference
fields (`availableWeekdays`, `targetMinutes`) are applied as *post-processing*
(reschedule + compress) on top of the engine output — they do not enter the
engine's internal split / exercise selection decisions.

## 4. Structured-action Coach Agent

The Agent's output space is bounded by a closed set of action types. Anything
outside this set is rejected by `output_validation` on the backend and again
by `LocalAgentActionExecutor` on Flutter.

Mutation actions (require confirmation, require trusted `sourceContextHash`):

- `generatePlan`
- `compressWorkout`
- `replaceExercise`
- `rescheduleWeek`

Non-mutating actions (no confirmation, no `sourceContextHash`):

- `weeklyReview`
- `nutritionAdvice`
- `safetyResponse`
- `answerOnly`

The full payload schema and per-action contract is in
[`docs/agent_capabilities.md`](agent_capabilities.md). The point for a
reviewer is that **the action space is small and explicit**, which is what
makes the rest of the safety story tractable.

## 5. Mutation lifecycle

The complete safe mutation lifecycle, from natural-language ask to local
write:

1. **User asks for a change** (e.g. "今天只有 30 分钟，帮我压缩训练").
2. **Agent proposes a structured action** (`compressWorkout` with
   `targetMinutes=30`).
3. **Mutation action requires confirmation** —
   `agent_backend/agents/action_safety.py` forces
   `requiresConfirmation=true` at the backend egress, overriding whatever the
   LLM filled in.
4. **Action preview shows before/after or summary** — Flutter's
   `AgentActionCard` + `AgentDiffView` render the diff against the current
   plan; nothing is written yet.
5. **`sourceContextHash` protects against stale plan context** — the hash is
   re-stamped by the backend from trusted server context (the request's
   `planContextHash`); Flutter rejects mutations whose hash no longer matches
   the current `planContextHash`.
6. **`LocalAgentActionExecutor` applies the change** only after the user
   clicks "应用修改" and the hash check passes; this is the *sole* AppState
   write entry on the agent path.
7. **`AppState` updates locally** and persists to `SharedPreferences`.

Every step is recoverable; if the user dismisses the card, nothing changed.

## 6. Safety and confirmation boundaries

Confirmation is one layer; deterministic safety is another. They sit in front
of the LLM, not behind it.

- **`safetyResponse` wins over mutation routing.** When a message hits the
  Chinese keyword guard in `agent_backend/safety/fitness_guardrails.py` (chest
  pain, dizziness, fainting, breathing trouble, pregnancy / eating-disorder
  risk, acute injury), the backend short-circuits to a deterministic
  `safetyResponse` **before** the LLM is called. The LLM never sees the
  message; no mutation action is ever produced.
- **High-risk symptoms do not route to plan mutation.** Even if the same
  message contains an explicit compression or reschedule intent, safety wins.
  This is verified by selected smoke cases (`recovery_high_risk_symptom_*`,
  `safety_recovery_chest_pain_dizzy_zh_008`).
- **No medical diagnosis.** The Agent does not diagnose injuries, conditions,
  or pain causes; `safetyResponse` advises stopping training and consulting a
  professional.
- **No injury diagnosis.** The Agent does not produce rehab prescriptions,
  exercise contraindications for specific pathologies, or return-to-sport
  timelines.
- **No fabricated recovery data.** The Agent never invents fatigue, soreness,
  sleep, HRV, PRs, body metrics, or injury data not present in the provided
  context. Empty `recentSessions` triggers a "数据不足" fallback rather than
  a confident summary.
- **`weeklyReview` remains non-mutating.** It is `requiresConfirmation=false`,
  carries no `sourceContextHash`, and `LocalAgentActionExecutor` treats it as
  a noop write-wise. The structured payload renders a read-only insight
  panel.

## 7. Recovery-routing phase evidence

The recovery-routing phase (PRs #43–#54) added recovery-aware coaching on top
of the existing structured-action infrastructure. It did **not** add new
mutation action types, new wearable signals, or any auto-execution path. The
behavioral additions were:

- **Recovery-aware `weeklyReview` suggestions** — high streak observations,
  completed-vs-weekly-frequency observations, suggestion-only footer, and a
  no-data fallback.
- **Explicit recovery compression routing to `compressWorkout`** — when the
  user combines a recovery context with explicit compression intent **and** a
  concrete number of minutes, the Agent routes to the existing
  `compressWorkout` mutation. Vague recovery questions ("要不要休息一下")
  remain non-mutating.
- **Explicit weekly availability reschedule routing to `rescheduleWeek`** —
  when the user combines recovery context with explicit weekly-availability
  intent and concrete weekday targets, the Agent routes to the existing
  `rescheduleWeek` mutation. `rescheduleWeek` adjusts weekly *available
  training days*; it does **not** represent moving one specific session from
  today to tomorrow.
- **Vague recovery questions stay non-mutating.** They get `answerOnly` or
  `weeklyReview`, never a silent mutation.
- **True today→tomorrow single-session movement is not currently
  supported.** It is intentionally non-mutating; future work in that
  direction would be a separate design proposal, not an extension of
  `rescheduleWeek`.

## 8. Eval and real-provider evidence

The Agent's behavior is locked by a three-layer eval contract:

- **`agent_backend/evals/coach_agent_eval_cases.json`** is the eval contract.
  Current counts: **58 total / 54 active / 4 expectedGap**. The 4
  `expectedGap` cases are kept on purpose as regression signals; they are not
  failures to "fix" by loosening the mock router.
- **`tests/test_coach_agent_mock.py`** locks the specific values the mock
  keyword router produces.
- **Pydantic `extra="forbid"`** and `tests/test_output_validation.py` lock
  the action / payload schema.

Four sanitized real-provider scorecards live in
[`docs/real_llm_scorecards/`](real_llm_scorecards/):

- `2026-05-09_mimo-v25-pro_smoke.md` — broad first-touch smoke (20/20).
- `2026-05-10_mimo-v25-pro_recovery-routing-smoke.md` — E-2 selected
  recovery-routing smoke (7 total / 6 pass / 1 fail).
- `2026-05-11_mimo-v25-pro_recovery-routing-smoke-after-e3.md` — E-4 same
  selection, after the prompt-first hardening of structured `weeklyReview`.
- `2026-05-12_mimo-v25-pro_recovery-compress-focused-rerun.md` — E-5 5×
  focused rerun of the one regressed compression case (5/5 pass).

These scorecards are **manual diagnostic evidence only** for narrow case sets
at specific points in time. Provider status remains experimental throughout;
none of the four scorecards is, individually or collectively, evidence of
production-readiness or a basis for provider promotion.

The selected-case real-provider harness now supports `--case-id` (repeatable)
and `--case-list` (comma-separated), so future selected smoke runs target
eval cases directly from the canonical eval file without temporary JSON. See
[`docs/real_llm_eval_harness.md`](real_llm_eval_harness.md) → *Selected case
runs*.

## 9. What this does not claim

This project, and this walkthrough, does **not** claim:

- production readiness;
- provider promotion (MiMo v2.5 Pro remains experimental);
- provider comparison (no second provider was evaluated);
- real-provider CI readiness (the manual harness intentionally stays off
  per-PR CI);
- autonomous medical coaching;
- autonomous plan mutation (every mutation continues to require user
  confirmation);
- true single-session movement support (today→tomorrow single-session moves
  remain non-mutating; a future feature would require a separate design
  proposal, not an overload of `rescheduleWeek`).

If any committed text in this repo implies one of the above, treat it as a
bug and file an issue.
