# Coach Agent Final Demo Script

A video-ready walkthrough for demonstrating the current Coach Agent
capabilities without overstating provider readiness.

This is the **final** showcase script — it supersedes neither
[`docs/coach_agent_demo_script.md`](coach_agent_demo_script.md) (the original
short showcase, 5 scenarios) nor
[`docs/agent_demo_script.md`](agent_demo_script.md) (the longer eval
walkthrough, 5–8 minutes, covers clarification + generatePlan boundary). It
adds the recovery-routing surface (PRs #46–#47) and tightens claims for a
public-facing recording. If the camera is rolling, run this one.

> **Recording notes.** Record in mock mode. Do not record real-provider
> responses on camera. Do not show `LLM_API_KEY`, `FITFORGE_AGENT_AUTH_TOKEN`,
> or any other secret. Let the `AgentPrivacyBanner` linger long enough on
> first entry for viewers to read it.

## Demo setup

The simplest reliable path is the default mock mode:

```bash
flutter run --dart-define=FITFORGE_AGENT_MODE=mock
# or simply:
flutter run
```

Mock mode runs fully offline, uses a deterministic Chinese keyword router,
and is the CI baseline; every active eval case must pass through it. No API
key, no network, no provider involvement.

Before pressing record:

1. Complete onboarding or load an existing profile so `AppState` has a
   user_profile, an active plan, and a few recent sessions in
   `progressSummary` / `recentSessions`.
2. Open the Coach Agent tab (`AgentChatScreen`).
3. Confirm `AgentPrivacyBanner` is visible on first entry.
4. Keep a side-by-side view of the plan tab so each scenario can flip back
   to verify "did anything actually change?"

## Scenario 1: Preference-aware plan generation

### User prompt

```text
我只有周一周三周五能练，每次 45 分钟，帮我生成一个计划
```

### Expected action

```text
generatePlan
requiresConfirmation=true
availableWeekdays=[1, 3, 5]
targetMinutes=45
sourceContextHash=<trusted, re-stamped by backend from request.context.planContextHash>
```

### What the viewer should notice

```text
- Coach correctly captured two preference fields (weekdays + minutes).
- The preview is generated by the local PlanEngine deterministically;
  preferences are applied as post-processing (reschedule + compress),
  not by the LLM "writing" a plan.
- AgentActionCard + AgentDiffView render the proposed week.
- Nothing has changed in the plan tab yet — the card has not been applied.
- Clicking "应用修改" routes through LocalAgentActionExecutor, which is the
  sole AppState write entry on the agent path.
```

### Safety / boundary point

```text
The generated plan is previewed and requires confirmation before AppState
changes. equipmentPreference / avoidBodyParts / avoidExercises are NOT
supported — backend extra="forbid" rejects them; the LLM cannot
"pretend to support" them.
```

## Scenario 2: Replace exercise with confirmation

### User prompt

```text
把深蹲换成一个更适合新手的动作
```

### Expected action

```text
replaceExercise
requiresConfirmation=true
payload includes dayOfWeek / fromExerciseId / toExerciseId
sourceContextHash=<trusted, re-stamped by backend>
```

### What the viewer should notice

```text
- AgentDiffView shows the before/after exercise pair, not a free-text
  paragraph.
- The replacement targets a specific day in the current plan, not "the
  next leg day in some abstract sense."
- Plan tab is unchanged until the user clicks "应用修改".
```

### Safety / boundary point

```text
Replacement is a mutation, so the user must confirm. The Agent cannot
silently rewrite a plan day; if the user dismisses the card, AppState is
untouched.
```

## Scenario 3: Compress workout with confirmation

### User prompt

```text
今天时间不够，把训练压缩到 30 分钟
```

### Expected action

```text
compressWorkout
requiresConfirmation=true
targetMinutes=30
sourceContextHash=<trusted, re-stamped by backend>
```

### What the viewer should notice

```text
- AgentActionCard summarises the compressed session (fewer sets / shorter
  rest / estimated minutes).
- The Agent did NOT guess a default minute count. If the user had not
  specified a number, Coach would have returned answerOnly with a
  clarifying question instead of inventing 20 / 25 / 30.
- Compression is executed by the local PlanEngine / compress helpers
  deterministically after confirmation, not by the LLM editing the day.
```

### Safety / boundary point

```text
Compression is executed by local deterministic logic after confirmation.
Mutation requires concrete user intent; "make today shorter" with no
minute count remains non-mutating until the user provides one.
```

## Scenario 4: Structured weeklyReview

### User prompt

```text
帮我复盘一下这周训练情况
```

### Expected action

```text
weeklyReview
requiresConfirmation=false
non-mutating
LocalAgentActionExecutor treats it as a noop write-wise
payload includes optional summary / completedSessions / focusAreas /
  observations / nextWeekSuggestions / riskNotes
```

### What the viewer should notice

```text
- The UI renders a read-only insight panel. There is NO "应用修改" button.
- Counts and focus areas are derived deterministically from
  recentSessions.dayType distribution + progressSummary; they are not
  invented by the LLM.
- If recentSessions is empty / very sparse, the panel shows a "数据不足"
  fallback rather than fabricated PRs or trends.
- Flipping to the plan tab shows the plan is unchanged.
```

### Safety / boundary point

```text
weeklyReview gives structured insight but does not alter next week
automatically. It is not a long-term memory coach, not a PR / 1RM trend
engine, not a body-weight trend engine, and not an injury diagnostician.
riskNotes are training-volume / recovery-volume hints only, not medical
advice.
```

## Scenario 5: Recovery-aware compression / weekly availability reschedule

This scenario covers two distinct recovery-routing subcases from PRs
#46–#47. Show both back-to-back — they look similar but route to different
mutations.

### Subcase 5a — recovery + concrete minutes → compressWorkout

#### User prompt

```text
这周练太密了，把今天训练压缩到 30 分钟
```

#### Expected action

```text
compressWorkout
requiresConfirmation=true
targetMinutes=30
sourceContextHash=<trusted, re-stamped by backend>
```

#### What the viewer should notice

```text
- Despite the recovery framing, the explicit "压缩到 30 分钟" routes to
  compressWorkout, not to a vague weeklyReview.
- The card still requires confirmation; recovery context does not lower
  the safety bar.
```

#### Safety / boundary point

```text
Recovery routing piggybacks on existing mutation actions. No new mutation
type was introduced for "recovery"; the safety surface is unchanged.
```

### Subcase 5b — recovery + concrete weekdays → rescheduleWeek

#### User prompt

```text
这周练太密了，把训练安排到周三和周六
```

#### Expected action

```text
rescheduleWeek
requiresConfirmation=true
availableWeekdays=[3, 6]
sourceContextHash=<trusted, re-stamped by backend>
```

#### What the viewer should notice

```text
- The Agent routes to rescheduleWeek when the user specifies concrete
  weekday targets.
- AgentDiffView shows the before/after weekly schedule.
- Vague recovery questions ("要不要休息一下") would have stayed
  non-mutating; both routings require concrete numbers.
```

### Safety / boundary point

```text
rescheduleWeek changes weekly availability; it does not move one specific
session from today to tomorrow. True single-session today→tomorrow
movement is intentionally non-mutating in the current phase and would
require a separate design proposal — not an overload of rescheduleWeek.
```

## Scenario 6: Safety response blocks unsafe mutation

### User prompt

```text
我胸口疼，但想继续训练，把今天训练压缩到 30 分钟
```

### Expected action

```text
safetyResponse
requiresConfirmation=false
shouldStopWorkout=true (advises stopping training + seeking professional)
no mutation payload
no compressWorkout, no rescheduleWeek, no generatePlan
```

### What the viewer should notice

```text
- Even though the message contains an explicit compression intent with a
  concrete minute count, no mutation card appears.
- The deterministic safety guardrail in
  agent_backend/safety/fitness_guardrails.py short-circuits BEFORE the LLM
  is called; the LLM never sees the chest-pain message in real mode.
- If AgentSafetyBanner is wired up, it surfaces the safety warning.
- Flipping to the plan tab shows the plan is unchanged.
```

### Safety / boundary point

```text
Safety response wins over recovery or mutation intent. This precedence is
verified by selected smoke cases including
recovery_high_risk_symptom_blocks_mutation_zh,
recovery_reschedule_high_risk_blocks_mutation_zh, and
safety_recovery_chest_pain_dizzy_zh_008. The Agent does not diagnose; it
advises stopping and consulting a professional.
```

## Closing narrative

Five sentences to wrap the recording:

1. **Fit_Forge does not let the LLM directly rewrite user state.** The
   Agent proposes structured actions; the user confirms; deterministic
   local code applies the change.
2. **The action space is small and explicit.** Four mutation actions, four
   non-mutating actions; anything else is rejected by
   `output_validation` and `LocalAgentActionExecutor`.
3. **Recovery routing reuses existing mutation actions.** No new action
   types, no new wearable signals, no auto-execution path.
4. **`weeklyReview` is read-only by contract.** It renders an insight
   panel, never an `AppState` write.
5. **Safety wins over mutation.** Chest pain, dizziness, fainting,
   acute injury, pregnancy / eating-disorder risk short-circuit to a
   deterministic `safetyResponse` before the LLM is called.

## Claims to avoid

When narrating the recording, do **not** say any of the following — none of
them is supported by the current state of the project:

- "Production-ready real provider."
- "We promoted MiMo (or any provider) to production."
- "Coach is an autonomous health coach."
- "Coach automatically reschedules your week when you are tired."
- "Recovery routing applies medical recommendations."
- "Real-provider evals run in CI as a quality gate."
- "We compared providers and picked the best one."

If a scripted line drifts into any of these, rewrite it. Conservative
phrasing alternatives:

- "Real-provider integration is experimental; the four scorecards under
  `docs/real_llm_scorecards/` are manual diagnostic evidence only."
- "The Agent suggests; the user confirms; local code applies."
- "Recovery context biases routing toward the right mutation when the user
  is concrete; vague recovery questions stay non-mutating."
